## 1. 문제 정의

### 핵심 문제

개발자가 Bitnaneun Studio OS 프로젝트에 투입되었을 때, **바로 작업에 들어갈 수 없다.**

원인은 세 가지다.

1. **스펙 문서가 방대하다.** 1064줄짜리 최종 개발 문서가 존재하지만, 제품 기획서 성격이 강해서 "지금 이 컴포넌트를 어떻게 만들어야 하는가"에 대한 답이 바로 나오지 않는다.

2. **기술 스택 불일치.** 스펙 문서는 SwiftUI 멀티플랫폼을 권장했으나, 실제 프로젝트는 Next.js 16 + React 19 + Tailwind CSS 4 + next-intl 기반 웹앱이다. 스펙의 아키텍처 지침을 그대로 따를 수 없으며, 웹 기술 스택에 맞게 재해석이 필요하다.

3. **컨벤션 부재.** 디자인 교체가 가능한 구조가 핵심 원칙인데, 네이밍/폴더 구조/토큰 사용 규칙 등 명확한 컨벤션 없이 개발하면 디자인과 기능이 결합되어 교체 불가능한 구조가 된다.

### 문제가 해결되지 않으면

- 개발자마다 다른 구조로 코드를 작성하여 일관성이 깨진다.
- 디자인 교체 시 기능 코드까지 대수술이 필요해진다.
- 새 개발자 온보딩에 과도한 시간이 소요된다.
- 스펙과 실제 구현 사이의 간극이 점점 벌어진다.

## 2. 컨텍스트

### 왜 지금인가

- 프로젝트는 이미 Next.js 16 기반으로 초기 세팅이 완료된 상태다.
- i18n(next-intl), 기본 레이아웃, 라우팅 구조가 잡혀 있다.
- 본격적인 화면 구현에 들어가기 **직전**이므로, 지금 가이드를 만들어야 처음부터 올바른 구조로 개발할 수 있다.
- 나중에 만들면 이미 작성된 코드를 리팩토링해야 하므로 비용이 급격히 증가한다.

### 프로젝트 배경

- **프로젝트명:** Bitnaneun Studio OS
- **조직:** 빛나는 스튜디오 (디자인 스튜디오)
- **핵심 목적:** 바쁜 업무 중 업무 현황을 바로 파악할 수 있게 하는 운영 OS
- **통합 서비스:** Notion, Google Calendar, Google Drive, Gmail, Slack
- **15개 엔티티:** User, Project, Task, Decision, Message, FileAsset, Shipment, Estimate, EstimateVersion, Event, Contact, PresenceSnapshot, AIInsight, ExternalLink, SyncJob
- **기술 스택:** Next.js 16 (App Router) + React 19 + Tailwind CSS 4 + next-intl
- **사용자:** 대표, 운영 담당, 디자이너 (3~10명 규모)
- **권한:** Admin, Manager, Member

## 3. 솔루션 제안

**컨벤션/패턴 가이드 + 구현 가이드를 하나로 통합한 개발 가이드 문서**를 작성한다.

이 문서는 개발자가 "어떤 화면을 만들어야 하는가"와 "어떤 규칙으로 만들어야 하는가"를 한 곳에서 찾을 수 있게 한다.

### 문서 구조 (8개 Part)

| Part | 제목 | 목적 |
|------|------|------|
| Part 1 | 프로젝트 개요 | 목적, 사용자, 핵심 원칙 요약. 3분 안에 프로젝트를 이해하게 함 |
| Part 2 | 기술 스택 & 프로젝트 구조 | Next.js App Router 기반 폴더 구조, 의존성, 환경 설정 |
| Part 3 | 디자인 시스템 | 토큰 정의, 의미 색상 규칙, 컴포넌트 목록, 스킨 분리 방법 |
| Part 4 | 데이터 모델 & API | 15개 엔티티 필드, 상태값, 커넥터 범위, 타입 정의 |
| Part 5 | 화면별 구현 가이드 | 홈, 프로젝트, 할 일, 소통, 파일, 견적, 팀 각 화면의 요구사항과 데이터 흐름 |
| Part 6 | 문구 규칙 & i18n | 문구 원칙, 좋은/나쁜 예시, next-intl 사용 규칙 |
| Part 7 | 성능 기준 & 최적화 | 로드 시간 목표, 가상화, 지연 로딩 기준 |
| Part 8 | Sprint 계획 & 체크리스트 | 4개 Sprint 범위, 완료 기준, MVP 체크리스트 |

### 핵심 제약

- 스펙 문서의 기능/정보 구조를 충실히 반영하되, 기술 구현은 Next.js 웹앱 맥락으로 재해석한다.
- 디자인 교체 가능 구조를 보장하는 컨벤션을 명시한다.
- 문서 자체는 읽기 쉽고 검색 가능해야 한다. 개발 중 참조 문서로 사용할 것이므로.

## 5. 사용자 스토리

### US-1: 프로젝트 온보딩

```
As a 새로 합류한 프론트엔드 개발자,
I want to 개발 가이드 문서의 Part 1~2를 읽고 프로젝트의 목적, 구조, 기술 스택을 파악하고 싶다,
So that 30분 이내에 로컬 환경을 세팅하고 첫 번째 작업에 착수할 수 있다.
```

**수용 기준:**

```
AC-1.1
Given 개발자가 프로젝트에 처음 합류했을 때
When Part 1(프로젝트 개요)을 읽으면
Then 프로젝트의 목적, 대상 사용자 3종(대표/운영 담당/디자이너), 핵심 원칙 4가지(사용자/정보/기술/디자인)를 설명할 수 있다.

AC-1.2
Given 개발자가 Part 2(기술 스택 & 프로젝트 구조)를 읽었을 때
When 로컬 개발 환경을 세팅하면
Then 폴더 구조, 주요 의존성, App Router 라우팅 규칙을 이해하고 dev 서버를 실행할 수 있다.

AC-1.3
Given 개발자가 Part 2의 폴더 구조를 읽었을 때
When 새 파일을 생성해야 하면
Then 해당 파일이 어느 디렉토리에 위치해야 하는지 3초 이내에 판단할 수 있다.
```

### US-2: 컴포넌트 생성 시 컨벤션 참고

```
As a 프론트엔드 개발자,
I want to 새 컴포넌트를 만들 때 네이밍 규칙, 파일 구조, 토큰 사용 규칙을 참고하고 싶다,
So that 디자인 교체가 가능한 일관된 구조로 컴포넌트를 만들 수 있다.
```

**수용 기준:**

```
AC-2.1
Given 개발자가 새 정보 컴포넌트(예: ActionCard)를 만들어야 할 때
When Part 3(디자인 시스템)의 컴포넌트 목록을 참고하면
Then 해당 컴포넌트의 의미 이름, 역할, 사용 맥락을 확인할 수 있다.

AC-2.2
Given 개발자가 컴포넌트에 색상을 적용해야 할 때
When Part 3의 의미 색상 규칙을 참고하면
Then 직접 색상값(red, blue) 대신 의미 토큰(danger, accent)을 사용하며, 어떤 상황에 어떤 토큰을 쓰는지 판단할 수 있다.

AC-2.3
Given 개발자가 간격, 둥근 모서리, 그림자 값을 적용해야 할 때
When Part 3의 토큰 정의를 참고하면
Then 하드코딩 값 대신 spacing.sm, radius.md, shadow.card 등 토큰을 사용할 수 있다.

AC-2.4
Given 디자이너가 다크모드를 적용하기로 결정했을 때
When 기존 컴포넌트의 기능 코드를 수정하지 않고
Then 토큰 값만 변경하여 다크모드가 적용된다.
```

### US-3: 화면 구현 시 요구사항 확인

```
As a 특정 화면을 구현하는 프론트엔드 개발자,
I want to 해당 화면의 섹션 구조, 표시 항목, 데이터 흐름을 한 곳에서 확인하고 싶다,
So that 스펙 문서 1064줄을 전부 읽지 않고도 해당 화면을 정확히 구현할 수 있다.
```

**수용 기준:**

```
AC-3.1
Given 개발자가 홈 화면을 구현해야 할 때
When Part 5의 홈 화면 가이드를 참고하면
Then 8개 섹션(상단 요약, 지금 볼 것, 오늘 할 일, 기다리는 것, 오늘 한 일, 이번 주 일정, 막힌 것, 빠른 실행)의 구조, 포함/제외 항목, 카드 구조를 확인할 수 있다.

AC-3.2
Given 개발자가 프로젝트 상세 화면을 구현해야 할 때
When Part 5의 프로젝트 화면 가이드를 참고하면
Then 헤더 정보(프로젝트명, 상태, 단계, 담당자, 다음 마감, 최신본)와 8개 섹션(개요, 일정, 할 일, 결정 기록, 소통, 파일, 견적, 발송물)의 구조를 확인할 수 있다.

AC-3.3
Given 개발자가 화면에서 사용할 데이터를 확인해야 할 때
When Part 4의 데이터 모델을 참고하면
Then 해당 엔티티의 필드명, 타입, 상태값을 확인하고 TypeScript 타입을 정의할 수 있다.

AC-3.4
Given 개발자가 모바일 레이아웃과 데스크탑 레이아웃을 구현해야 할 때
When Part 5의 레이아웃 가이드를 참고하면
Then 모바일(단일 컬럼 + 하단 네비 5개)과 데스크탑(사이드바 + 중앙 + 우측) 구조를 정확히 구현할 수 있다.
```

AC-4.3

When 코드 리뷰에서 하드코딩된 색상/간격/둥근모서리 값이 발견되면
Then 해당 값을 어떤 토큰으로 대체해야 하는지 가이드에서 바로 찾을 수 있다.

## 8. 가이드 문서 상세 구조

아래는 실제 개발 가이드 문서에 들어갈 각 Part의 상세 목차다.

### Part 1: 프로젝트 개요

- 1.1 프로젝트 한 줄 정의
- 1.2 핵심 목표: 복잡한 업무를 쉬운 행동으로
- 1.3 사용자 유형 (대표 / 운영 담당 / 디자이너)
- 1.4 핵심 원칙 요약 (사용자 / 정보 / 기술 / 디자인)

## 9. 개발팀 주의사항 (스펙 문서 Section 27 기반)

1. 홈을 프로젝트 리스트로 만들지 말 것 -- 홈은 운영 상황판이다.
2. 모든 화면을 데이터 출처 기준으로 나누지 말 것 -- 의미 기준으로 나눈다.
3. 카드 하나에 너무 많은 텍스트를 넣지 말 것.
4. 이미 끝난 일과 기다리는 일을 오늘 할 일에 섞지 말 것.
5. 시각 표현에 상태 판단 로직을 박아넣지 말 것 -- 로직과 스타일 분리.
6. 문구를 보고서 말투로 쓰지 말 것 -- 일상어 사용.
7. 디자인 시스템 없이 바로 화면부터 찍지 말 것.
